# Implementation Plan: Commute Flow Explorer

**Branch**: `001-commute-flow-explorer` | **Date**: 2026-06-25 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `specs/001-commute-flow-explorer/spec.md`

---

## Summary

Build a client-side-only web app that lets transport planners click locations on a MapLibre
map, resolve clicks to H3 resolution-9 hexagonal clusters, query pre-processed LODES OD
Parquet files in-browser with DuckDB-WASM, and visualise commuter flows as Flowmap.gl
bidirectional arcs and an OD matrix table. A Python pipeline (uv-managed) pre-processes raw
LODES CSV data into Parquet artefacts committed to the repo; no server is ever needed at
runtime.

---

## Technical Context

**Language/Version**: JavaScript (ESM) / React 18; Python 3.11 (pipeline only)

**Primary Dependencies**:
- `maplibre-gl` v4 — map rendering
- `h3-js` v4 — hex resolution and k-ring computation
- `@flowmap.gl/core` — bidirectional arc layer
- `@duckdb/duckdb-wasm` — in-browser Parquet query
- `vite` + `@vitejs/plugin-react` — build tooling
- Pipeline: `pandas`, `pyarrow`, `h3` (≥4.0), `pyyaml`, `requests`, `tqdm`

**Storage**: Static Parquet files served from `public/data/`. No database server.
Git LFS for files ≥ 50 MB.

**Testing**: No automated test suite in v1. Validation via quickstart.md manual scenarios.

**Target Platform**: Modern desktop browser (Chrome/Firefox/Edge latest). Hosted on GitHub
Pages (static). No server-side runtime.

**Performance Goals**: First arc render ≤ 3 s on 10 Mbps (including Parquet download).
Year-switch with cached Parquet ≤ 1 s. DuckDB `IN`-clause query ≤ 2 s for ≤ 740 hex IDs.

**Constraints**: Zero server-side code. No API keys. Files ≥ 50 MB use Git LFS. Total
Parquet payload across 22 years: 110–450 MB (Git LFS). Per-session browser download: one
year file (~5–20 MB) + hex_meta (~2–4 MB).

**Scale/Scope**: ~20 max points per session. 22 years of data. 8 Utah counties (v1).

## Phase 0: Research Summary

See [research.md](./research.md) for full decision log. Key resolved questions:

- **DuckDB-WASM bundle**: MVPBundle (no SharedArrayBuffer, works on GitHub Pages)
- **AGRC style merging**: `resolveAGRCStyle()` utility fetches + rewrites relative URLs,
  merges three styles with source-key prefixes to avoid collisions
- **Flowmap.gl self-loops**: use `selfLoop` property; renders as ring indicator at centroid
- **h3-js cellToBoundary**: always pass `true` for GeoJSON `[lng, lat]` order
- **WAC join**: left join before hex aggregation; fillna(0) for missing work blocks
- **aux deduplication**: deduplicate on `(h_geocode, w_geocode)` before hex aggregation

## Phase 1: Design Decisions

### State Architecture (App.jsx)

```
App state:
  year:       number (default 2023)
  kRing:      number (default 1)
  activeView: 'map' | 'matrix'
  points:     Point[]           (managed by usePoints)
  flows:      ODFlow[]          (managed by useFlows)
  hexMeta:    Map<h3_id, meta>  (loaded once at startup)
```

`useFlows` subscribes to `points` + `year` + `kRing`; re-queries DuckDB when any changes.

### DuckDB Lifecycle

1. `initDB()` called once on mount — initialises DuckDB-WASM MVPBundle.
2. `hex_meta.parquet` fetched and parsed into JS Map (startup).
3. `loadYear(year)` called lazily when a year is first queried:
   `CREATE OR REPLACE VIEW od AS SELECT * FROM parquet_scan('/data/od_${year}.parquet')`
4. `queryOD(allHexIds)` executes the `WHERE h_h3 IN (...) AND w_h3 IN (...)` query.

### MapLibre Layer Order

1. AGRC merged style (background + base + labels)
2. `hex-cluster-fill` — GeoJSON fill layer, one feature per hex cell, `fill-color` from
   point colour via feature property
3. Flowmap.gl custom layer (arcs)
4. MapLibre Marker elements (point pins, rendered via DOM, not WebGL)

### OD Matrix Aggregation

`useFlows` maps each `(h_h3, w_h3)` raw flow row to a `(originPointId, destPointId)` pair
by looking up which point's clusterIds contain h_h3 and w_h3. Sums S000/SA01/SA02/SA03/
SI01/SI02/SI03 per matrix cell. A shared hex (overlap) contributes to both relevant cells —
consistent with the overlap warning.
